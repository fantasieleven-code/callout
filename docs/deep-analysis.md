## 一、三个视角的审查

### 1.1 CTO视角：「这东西技术上站得住吗？」

**结论：站得住，但要砍掉60%的想法。**

#### 肯定的部分

- **MCP是正确的切入点**。2025年12月MCP已捐给Linux Foundation下的AAIF，成为行业标准。Claude Code / Cursor / VS Code / Windsurf全部支持。**一个MCP Server = 覆盖所有主流工具**，这是2024年不存在的机会窗口。
- **不写代码是对的**。Archon做决策层，代码生成交给已有工具。这避免了和Cursor（$2B ARR）、Claude Code正面竞争。
- **多视角审查有技术可行性**。本质是5套专家Prompt + 项目上下文注入。不需要多Agent框架（CrewAI那种），一个MCP tool调5次LLM就行。

#### 要砍掉的

| 原方案 | CTO判断 | 理由 |
|--------|---------|------|
| 七步引擎全部自动化 | 砍。V1只做Step 6（多视角审查） | 七步是方法论不是软件。Step 1-5人脑做就行，Step 6是唯一机器能做得比人好的 |
| 健康度仪表盘 | 砍 | `npm test` + `npx tsc --noEmit` 已经够了，别造轮子 |
| 决策日志系统 | 简化为CLAUDE.md里一个表格 | 专门的YAML决策系统是over-engineering |
| 方法论模板市场 | V1不做 | 没有1000个用户之前没有"市场" |
| VS Code Extension | V1不做 | MCP已经跨平台了，Extension是重复工作 |
| Team版多人协作 | V1不做 | 先证明单人版有价值 |

#### CTO的V1定义

```
Archon V1 = 一个MCP Server + 5个审查Prompt + 1个上下文收集器

功能：
  /archon review → 读代码+架构+CLAUDE.md → 5个专家角色审查 → 输出报告
  /archon decide "决策内容" → 追加到决策表

就这些。别的都不做。
```

**开发工期：1人×1周**。不是4周。

### 1.2 创业团队负责人视角：「这能变成生意吗？」

**结论：能，但不是SaaS生意，是内容+工具生意。**

#### 市场判断

- **TAM（可触达市场）**：全球独立开发者~3000万，其中使用AI编码工具的约40%（~1200万）。但愿意为「方法论」付费的 < 1%。
- **真实的pain point**：不是「缺方法论」，是**「写了3天发现方向错了」**。用户不会搜「methodology tool」，会搜「how to avoid rewriting my project」「AI code review architecture」。
- **竞品壁垒低**：5个Prompt + MCP Server，技术门槛约3天。壁垒不在代码，在**品牌和案例库**。

#### 真正的商业模式

```
阶段1（0-6个月）：内容驱动 → 免费工具获客
  ├── 核心资产：CodeLens A+ 从零到一的完整案例
  ├── 内容分发：Twitter/X + 博客 + YouTube/B站
  ├── 工具：开源MCP Server（免费）
  └── 收入：$0（投入期）

阶段2（6-12个月）：咨询驱动 → 工具是获客漏斗
  ├── 产品：「AI架构审查即服务」— 人工+AI混合
  ├── 定价：$500-2000/次审查（不是$29/月SaaS）
  ├── 客户：种子轮/A轮创业公司的CTO
  └── 预期：10-20个客户/月 = $5K-40K/月

阶段3（12-24个月）：根据阶段2数据决定
  ├── 如果咨询做得好 → 做成SaaS自动化
  └── 如果咨询做不起来 → 方向错了，止损
```

**关键认知：不要上来就做SaaS。先用咨询验证市场，用工具降低咨询成本。**

类比：PostHog的CEO说过「open source is a great initial distribution channel, but doesn't make for great monetization」。免费工具 = 分发渠道，不是收入来源。

### 1.3 想做软件创业的用户视角：「我会用这个吗？」

**结论：看到「方法论」三个字就关了。但如果叫「AI架构审查」就有兴趣。**

#### 用户真实心理

```
❌ 用户不想要的：

✅ 用户真正想要的：
  「帮我看看这个架构有没有坑」— 即时价值
  「告诉我哪些功能可以砍掉」— 省时间
  「从CTO/安全/客户角度审查一下」— 自己想不到的视角

#### 用户体验重新设计

```

### 4.2 关键发现

**没有任何一个工具做「多视角审查」**。

- Spec-Kit最接近我们的理念（结构化方法论），但它是**Prompt模板集**，不是Agent，不做审查
- MetaGPT有多角色，但是**生成代码**的角色，不是**审查代码**的角色
- APM有方法论，但是**项目管理**方法论，不是**架构审查**方法论
- 所有工具都在帮「怎么做」，没有工具帮「做得对不对」

### 4.3 最大威胁：GitHub Spec-Kit

```
Spec-Kit (74.4K stars) vs Archon：

Spec-Kit 的优势：
  ✅ GitHub官方出品，自带流量
  ✅ 已覆盖20+工具
  ✅ 结构化流程（specify→plan→tasks→implement）
  ✅ MIT开源

Spec-Kit 的弱点（= 我们的机会）：
  ❌ 纯Prompt模板，不是Agent — 用户要手动触发每一步
  ❌ 没有审查功能 — 只管往前走，不管走对没有
  ❌ 没有上下文收集 — 不会自动读你的代码和决策
  ❌ 没有角色切换 — 不会从CTO/安全/客户视角挑战你

我们的差异化定位：
  「Spec-Kit帮你规划怎么做。Archon帮你检查做得对不对。」
  → 不竞争，互补。甚至可以在Spec-Kit流程末尾调用Archon审查。
```

### 4.4 APM（Agentic Project Management）— 值得关注

```
APM (1.7K stars) vs Archon：

相似点：
  - 都强调方法论驱动
  - 都支持多工具（APM支持10+，我们通过MCP）
  - 都有任务分解和依赖追踪

差异：
  APM = 项目管理（怎么分任务、怎么跟进度）
  Archon = 架构审查（做得对不对、该砍什么该补什么）

  APM解决：「AI怎么按步骤完成一个大任务」
  Archon解决：「这个任务本身是不是应该做」

结论：不是竞品，是互补。APM管执行，Archon管方向。
```

## 五、如何接入主流编码工具

### 5.1 MCP = 唯一需要做的集成

2026年的关键发现：**MCP已经是行业标准**，一个Server覆盖所有工具。

| 工具 | MCP配置文件 | 传输协议 |
|------|------------|---------|
| Claude Code | `.mcp.json` / `~/.claude.json` | stdio / HTTP / SSE |
| Cursor | `.cursor/mcp.json` | stdio / HTTP / SSE |
| VS Code / Copilot | `.vscode/mcp.json` | stdio / HTTP |
| Windsurf | `~/.codeium/windsurf/mcp_config.json` | stdio / HTTP / SSE |

**所有4个工具用同一个 `mcpServers` JSON结构。一次开发，四处运行。**

### 5.2 技术实现方案

```
选择：Streamable HTTP MCP Server（不是stdio）

原因：
  stdio = 本地子进程，每个工具独立启动
  HTTP = 网络服务，所有工具连同一个端点

  → HTTP更适合「远程审查」场景（审查别人的项目）
  → HTTP也支持本地运行（localhost）
  → 未来做SaaS时天然支持云端部署
```

**最小实现：**

```typescript
// archon-server.ts — 完整MCP Server
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { z } from "zod";

const server = new McpServer({ name: "archon", version: "0.1.0" });

// 核心工具：多视角审查
server.tool(
  "review",
  "对当前项目进行多视角架构审查（CTO/安全/产品/运维/客户）",
  {
    perspectives: z.array(z.enum(["cto", "security", "product", "devops", "customer"]))
      .optional()
      .describe("指定审查视角，默认全部"),
    customer_role: z.string().optional()
      .describe("自定义客户角色，如'HRVP'或'CEO'"),
  },
  async ({ perspectives, customer_role }) => {
    // 1. 收集项目上下文（代码结构、CLAUDE.md、package.json、测试结果）
    // 2. 对每个视角调用对应的专家Prompt
    // 3. 汇总输出报告
    return { content: [{ type: "text", text: report }] };
  }
);
```

### 5.3 分发方式

```
npm install -g archon-dev        # CLI安装
npx archon-dev serve              # 启动HTTP MCP Server

# Claude Code
claude mcp add archon -- npx archon-dev serve

# Cursor — 在 .cursor/mcp.json 中添加
{
  "mcpServers": {
    "archon": { "command": "npx", "args": ["archon-dev", "serve"] }
  }
}

# VS Code — 在 .vscode/mcp.json 中添加（同样格式）
```

MCP覆盖所有工具。同时做一个Skill让Claude Code用户体验更好：

你是Archon架构审查引擎。对当前项目执行5个专家视角的审查。

## 上下文收集
!`find . -name "*.ts" -not -path "*/node_modules/*" | head -50`
!`cat CLAUDE.md 2>/dev/null | head -100`

## 六、开源 vs 收费 — 深度分析

### 6.1 核心问题：开源能不能收费？

**能。2026年最赚钱的开发者工具大多是开源的。**

| 公司 | 开源 | ARR | 模式 |
|------|------|-----|------|
| GitLab | CE开源 / EE付费 | ~$992M | 开源核心 + 企业授权 |
| Sentry | BSL开源 | $128M | 开源 + 用量付费云 |
| Supabase | 全部开源 | $70M | 开源 + 托管云 |
| n8n | Fair-code开源 | $40M | 开源 + 企业授权 + 云 |
| PostHog | MIT开源 | 数千万 | 开源 + 用量付费云 |
| Cursor | **不开源** | $2B | 纯SaaS |

**结论**：开源不仅能收费，而且开源是最有效的分发渠道。PostHog CEO原话：「open source is a great initial distribution channel」。

### 6.2 Archon应该开源还是收费？

**开源。原因如下：**

| 维度 | 开源 | 闭源SaaS |
|------|------|---------|
| 分发 | GitHub星标 + npm = 免费流量 | 需要投广告/做SEO |
| 信任 | 「代码都给你看了」→ 开发者信任 | 「你的代码发给你的服务器？」→ 抵触 |
| 社区 | PR贡献新的审查视角/模板 | 所有功能自己做 |
| 壁垒 | 品牌+社区+案例库 | 代码（3天可复制） |
| 定价压力 | 企业功能收费合理 | $29/月对比免费替代品 |

**核心逻辑**：Archon的代码量很小（MCP Server + 5个Prompt），闭源没有意义。壁垒不在代码，在品牌和方法论深度。

### 6.3 推荐许可证：AGPL v3 + 商业双授权

```
AGPL v3 的特性：
  ✅ OSI认证的正式开源许可证（可以叫自己「开源」）
  ✅ 网络传播条款：如果别人把Archon做成SaaS，必须开源全部代码
  ✅ 企业客户如果不想被AGPL传染 → 买商业授权（= 收入来源）
  ✅ 个人/小团队自用完全免费

谁用AGPL：
  - MongoDB（早期）
  - Grafana
  - Plausible Analytics
  - n8n（部分）
```

**为什么不用MIT**：MIT允许任何人（包括大厂）拿你的代码做商业服务不给一分钱。Spec-Kit用MIT，但它背后是GitHub，不需要靠代码赚钱。我们需要。

**为什么不用BSL**：HashiCorp用BSL后新客户增长暴跌至1.5% QoQ，社区分叉出OpenTofu。BSL会让社区敌视。
