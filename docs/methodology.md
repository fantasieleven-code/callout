# 从零到一：AI产品单人全栈建设方法论

> 提炼自 CodeLens A+ v3.0 的完整建设过程（2026-03-01 ~ 2026-03-06，6天，1人+AI协作）。
> 适用场景：技术创始人/独立开发者从概念到可演示产品的全流程。

---

## 全景路线图

```
概念验证 → 框架设计 → 架构审查 → 技术Spike → 分阶段实现 → 多视角验证 → 商业化补齐
  Day 1       Day 1      Day 2       Day 2      Day 3-5        Day 5         Day 6
```

---

## 第一阶段：概念锚定（Day 1 上午）

### 方法：「工作场景正推法」

不从功能出发，从真实工作场景出发。每个设计决策必须回答：

> 「这对应用户/客户实际工作中的什么场景？回答不了就不做。」

#### 我们做了什么

1. **定义核心问题**：AI时代，工程师的哪些能力是AI替代不了的？
2. **推导评估维度**：从「入职后第一周会遇到什么」反推出5个能力维度（V1-V5）
3. **设计评估流程**：从「真实工作节奏」反推出70min连续项目+4 checkpoint
4. **明确不做清单**：Token经济学、YAML约束、AGENTS.md写作 — 这些是「考试技巧」不是「工程能力」

#### 可复用原则

| 原则 | 含义 | 反例 |
|------|------|------|
| 场景正推 | 每个功能对应一个真实用户动作 | 「加个设置页因为别人都有」 |
| 信号>指标 | 少量高信噪比数据优于大量弱信号 | 采集50种行为但没一个准的 |
| 渐进式自动化 | V1做70%自动就够，不追求100% | 花3周做全自动但没人用 |

#### 交付物

- 考核框架文档（维度定义 + 权重 + 衡量标准）
- 面试流程文档（时间线 + checkpoint规格 + 题库模板）
- **不做清单**（和做什么同等重要）

## 第二阶段：架构设计 — 三层决策法（Day 1 下午）

### 方法：「复用-重构-新增」三分法

不做全新设计，先盘点手头有什么。

```
已有资产盘点
    ├── 可直接复用（~55-60%）→ 接口不变，换皮/换配置
    ├── 需要重构（~25-30%）  → 核心逻辑保留，接口/流程重做
    └── 必须新增（~15-20%）  → 从零写，但控制在最小范围
```

#### 我们做了什么

| 分类 | 资产 | 决策理由 |
|------|------|---------|
| 复用 | React+Monaco / Express+Socket.IO / PostgreSQL+Redis / E2B / JWT Auth | 已验证，换了反而有风险 |
| 重构 | 面试流程引擎、AI Prompt、评分算法、前端布局 | 业务逻辑全变，技术栈不变 |
| 新增 | 行为信号SDK、AI追问引擎、诚信度报告 | 核心差异化，必须从零做 |

### 方法：「设计决策表」

每个技术选型记录为一行：**决策 | 选择 | 理由**。

这张表的价值：
1. 新成员看完就知道「为什么这么做」
2. 避免重复讨论已确认的决策
3. 未来回溯时能追踪每个选择的上下文

我们的决策表最终有 **30+ 条**，涵盖面试结构、AI模型选型、RTC方案、存储策略等。

#### 交付物

- 技术架构文档（系统架构图 + API设计 + 数据库Schema）
- 设计决策表（30+条，每条有选择+理由）
- 开发计划（里程碑 + 优先级 + 风险 + 工期）

## 第三阶段：架构审查 — 「找漏洞而非找亮点」（Day 2 上午）

### 方法：四维度全面审查

不是走过场式review，是刻意寻找问题。

```
架构审查四维度：
├── 工程维度：性能瓶颈、单点故障、可扩展性
├── 架构维度：耦合度、状态管理、数据一致性
├── 业务流程维度：异常路径、边界条件、容错
└── 可维护性维度：测试覆盖、日志可观测、部署复杂度
```

#### 我们做了什么

一次性产出 **AR-1 ~ AR-15**（15个架构待办项），按优先级分为：

| 优先级 | 数量 | 含义 |
|--------|------|------|
| P0 Blocker | 2 | 不解决=上不了线 |
| P1 Critical | 3 | 不解决=核心体验差 |
| P2 Important | 6 | 尽快解决 |
| P3 Nice-to-have | 4 | 可延后 |

关键发现：

- **AR-1 AI上下文窗口管理**：面试官对话会爆context → 设计渐进式摘要
- **AR-2 文件同步一致性**：Monaco↔E2B双向同步会冲突 → 设计版本号+server-wins
- **AR-7 E2B单点故障**：沙箱挂了整个面试废了 → 设计mock PTY降级
- **AR-9 反作弊信号**：等面试完再采集就晚了 → 前端实时采集+批量上传

#### 可复用原则

> **审查时机**：设计完、编码前。不是编码完再审查（那时改不动了）。

> **审查心态**：假设每个组件都会挂，问「挂了怎么办」。

#### 交付物

- 架构审查文档（15个待办项 + 优先级 + 建议解决阶段）
- 待确认清单（PENDING-1 ~ PENDING-13，区分阻塞/非阻塞）

## 第四阶段：技术Spike — 「花1天验证，省3周返工」（Day 2 下午）

### 方法：风险驱动的定向验证

不是写demo玩，是针对「如果验证失败，整个架构要推翻重来」的点做验证。

#### 我们做了什么

| Spike | 验证内容 | 耗时 | 结论 |
|-------|---------|------|------|
| SP-1 | E2B能否支持持久文件系统+PTY+自定义模板 | 2h | 通过，注意PTY要设`timeoutMs:0` |
| SP-2 | Monaco↔E2B双向同步方案 | 2h | 全量序列化+500ms debounce（500行项目单文件仅3-5KB，diff不值得） |
| SP-3 | AI上下文窗口是否够用 | 2h | 峰值17K tokens，32K窗口够用，单场成本$0.12-0.20 |
| SP-4 | 前端5个Store怎么划分 | 1h | Zustand 5 Store + Orchestrator |
| SP-5 | RTC+Monaco+xterm.js能否共存 | 1h | 零冲突，4个完全不同传输层 |

#### 可复用原则

> **Spike选择标准**：「如果这个验证失败，我要重新选型/重新设计吗？」是→做Spike。否→直接编码。

> **Spike输出标准**：不是代码，是**结论+数据+工程注意事项**。比如SP-2不只是说「能同步」，而是精确到「echo suppression用2s TTL Map」。

## 第五阶段：分阶段实现 — 「TODO驱动开发」（Day 3-5）

### 方法：里程碑分解 + 自动化任务推进

#### 5.1 里程碑分解原则

```
Phase 1: 基础设施（2周）  — 能跑起来
Phase 2: 核心流程（4周）  — 核心功能可用
Phase 3: 信号与评分（3周） — 数据闭环
Phase 4: 反作弊（2周）    — 防御层
Phase 5: 管理端（2周）    — 客户可用
Phase 6: 集成部署（2周）  — 可上线
```

关键原则：**每个Phase结束时都有可演示的增量**，不是Phase 6才能看到东西。

#### 5.2 任务编排模式

每个Phase内部拆成 **M（Milestone）→ Task → Subtask**：

```
M0-1 项目初始化
  ├── Monorepo搭建
  ├── TypeScript strict
  ├── Docker Compose
  └── ...

M1-1 Checkpoint流程引擎
  ├── 状态模型定义
  ├── 计时器管理
  ├── 数据快照
  └── 异常处理
```

#### 5.3 「测试伴生」原则

**每写一个功能，同步写测试。不是写完再补测试。**

```
编码循环：
  写功能代码 → 写测试 → 跑测试 → 通过 → 下一个功能
                         ↓ 失败
                    修复 → 重跑
```

我们的测试数据增长：

| 阶段 | 累计测试数 | 新增 |
|------|-----------|------|
| Phase 1 完成 | ~180 | 180 |
| Phase 2 完成 | ~380 | 200 |
| Phase 3 完成 | ~500 | 120 |
| Phase 4 完成 | ~560 | 60 |
| Phase 5 完成 | ~620 | 60 |
| 商业化补齐 | 691 | 71 |

测试类型分布：

| 类型 | 数量 | 覆盖内容 |
|------|------|---------|
| 单元测试（Server） | ~530 | Services + Routes + WS + Middleware |
| 单元测试（Client） | ~160 | Stores + Components + Hooks |
| E2E测试 | 7 specs | 面试流程 + 管理后台 + 断线恢复 + 候选人加入 |

#### 5.4 编码节奏

实际执行节奏（单人+AI协作）：

| 时间段 | 内容 | 产出 |
|--------|------|------|
| Day 3 上午 | Phase 1: M0-1~M0-6 基础设施 | 项目骨架 + DB + Auth + WS + Sandbox + AI Router |
| Day 3 下午 | Phase 2: M1-1~M1-3 Checkpoint + 模板 + IDE | 面试流程引擎 + 前端IDE |
| Day 4 上午 | Phase 2: M1-4~M1-7 AI面试官 + 语音 | AI Prompt + 追问 + RTC双模式 |
| Day 4 下午 | Phase 3: M2-1~M2-8 信号采集 + 评分 | 5维度评分引擎 + 行为SDK |
| Day 5 上午 | Phase 4: M3-1~M3-3 反作弊 | 信号分析 + 诚信度报告 |
| Day 5 下午 | Phase 5: M4-1~M4-3 + Phase 6 管理端 + 部署 | 13个admin页面 + Docker + CI |

#### 5.5 「架构审查项嵌入」策略

AR-1~AR-15 不是单独做的，而是**嵌入到对应Phase中一并处理**：

```
Phase 1 中处理：AR-1(上下文管理), AR-2(文件同步), AR-3(测试体系)
Phase 2 中处理：AR-5(容错), AR-6(错误处理), AR-7(降级), AR-8(状态恢复), AR-9(反作弊信号)
Phase 3 中处理：AR-10(Prompt版本化), AR-11(多模板), AR-13(API文档)
Phase 4+ 中处理：AR-12(环境安全), AR-14(组件库), AR-15(EventLog优化)
```

这样做的好处：不会出现「功能做完了，回头发现架构问题要大改」。

## 第六阶段：多视角验证 — 「用不同的眼睛看同一个产品」（Day 5 晚）

### 方法：角色切换式审查

不是自己审查自己，是切换到完全不同的角色视角，问不同的问题。

#### 6.1 第一轮：技术视角三合一

同时扮演三个角色审查架构：

| 角色 | 关注点 | 关键发现 |
|------|--------|---------|
| **CTO** | 架构合理性、技术债、扩展性 | 角色系统过度设计（4级→2级），JWT别改结构（加字段），Landing Page不需要SEO |
| **HR SaaS专家** | 产品-市场匹配、用户习惯 | 「HR不会自注册」→ 砍掉自注册，改为白手套服务；「别展示定价」→ 改为"联系我们" |
| **DevOps专家** | 可部署性、运维复杂度 | 先解决「能跑」，计费系统不需要DB表（用config对象），别过早做K8s |

**关键产出**：原计划9天的商业化工作 → 砍到3.5天。**砍掉的比做了的更重要。**

#### 6.2 第二轮：客户视角审查

切换到产品的**购买者**视角：

| 角色 | 核心诉求 | 发现的缺失 |
|------|---------|-----------|
| **HRVP（使用者）** | 效率、协作、数据 | 缺ATS对接、缺报告分享、缺批量导入、缺PDF导出、缺候选人反馈 |
| **CEO（决策者）** | ROI、合规、安全 | 缺合规声明、缺数据安全说明、缺ROI指标 |

**关键产出**：6个紧急补齐项（报告分享 → PDF → 批量导入 → 合规页 → 候选人反馈 → ATS Webhook）。

#### 6.3 验证的执行模式

```
多视角验证
    ├── 技术三视角 → 砍掉过度设计 → 精简TODO
    ├── 客户双视角 → 发现体验缺口 → 补齐TODO
    └── 按优先级执行补齐 → 每个补齐项带测试
```

#### 可复用原则

> **审查频率**：不是最后才做。每完成一个大阶段（Phase 2完/商业化前/上线前）都做一轮。

> **审查方法**：不是问「做得好不好」，是问「如果我是X角色，我会拒绝买/用这个产品的原因是什么」。

> **审查产出**：不是意见列表，是**排好优先级的TODO**，直接进入执行。

## 第七阶段：商业化补齐 — 「最小可售卖」（Day 6）

### 方法：客户旅程倒推

不是从技术功能出发，是从客户使用旅程出发：

```
客户旅程：
  看到产品 → 想了解 → 联系我们 → 看Demo → 创建账户 → 导入候选人 →
  发面试 → 候选人完成 → 看报告 → 分享报告 → ATS收到通知 → 候选人给反馈
```

每一步检查：有没有对应的功能？缺的就是要补的。

## 横切关注点：贯穿全程的方法

### A. 测试策略：「测试伴生，不是测试补课」

```
每个功能的交付标准：
  ✅ 功能代码
  ✅ 单元测试（happy path + edge cases）
  ✅ TypeScript编译通过
  ✅ 全量测试跑过（不引入回归）
```

测试金字塔：

```
        /  E2E (7)  \          — 验证用户旅程
       / Integration \         — 验证模块间交互
      /  Unit (691)    \       — 验证单个函数/组件
```

### B. 文档策略：「决策记录 > 使用说明」

我们维护的文档：

| 文档 | 内容 | 更新频率 |
|------|------|---------|
| CLAUDE.md | 全量TODO + 决策表 + 进度追踪 | 每个Phase更新 |
| 01-考核框架.md | 维度定义（稳定后几乎不变） | 一次性 |
| 02-面试流程.md | 流程规格（稳定后几乎不变） | 一次性 |
| 03-技术架构.md | 系统架构 | Phase 1后稳定 |
| 04-开发计划.md | 里程碑计划 | 执行中微调 |
| 05-架构审查.md | 审查结论 + Spike报告 | Day 2后稳定 |

关键原则：**记录「为什么这么做」比记录「怎么做」更重要**。代码会说how，文档要说why。

### C. 依赖管理：「标注不可做项」

不是所有事情都能做。关键是**明确标注**每个不可做项需要什么外部资源：

```
✅ 可做：代码、测试、文档、架构设计
⚠️ 不可做 + 标注原因：
  - Pilot测试 → 需要5名测试工程师
  - 评分校准 → 需要Pilot数据
  - CustomLLM部署 → 需要VOLC_AK/SK凭证
  - 压力测试 → 需要真实DB+E2B环境
  - 监控/备份 → 需要Grafana Cloud + Sentry DSN
```

这样做的好处：交接时对方立刻知道「接下来需要准备什么」。

### D. 知识产权：「概念用专利，参数用秘密」

在产品公开前评估IP保护：

| 保护方式 | 适用对象 | 示例 |
|---------|---------|------|
| 发明专利 | 方法论创新 | AI幻觉注入式验证、行为认知画像 |
| 实用新型 | 技术实现 | 偏差仲裁评分、调试行为分类 |
| 商业秘密 | 参数/数据 | 评分prompt、权重矩阵、幻觉库 |

## 反思：什么有效，什么可以更好

### 有效的做法

1. **架构审查前置**：Day 2做的审查，省掉了后面至少3次「发现架构问题要回头改」
2. **Spike验证**：SP-2发现「500行项目不需要增量diff」，避免了花2天实现不需要的OT算法
3. **多视角验证**：HR SaaS专家一句「HR不会自注册」砍掉了2天的自注册系统开发
4. **测试伴生**：691个测试从没出现过「改了A坏了B」的回归
5. **不做清单**：明确不做的比做了的更重要，避免了scope creep

### 可以更好的地方

1. **Pilot数据缺失**：评分阈值、校准、区分度全部依赖未来的Pilot数据。理想情况应该先做Pilot再开发
2. **前端E2E覆盖不足**：7个spec不够，尤其缺少完整70min面试流程的自动化测试
3. **性能基准缺失**：没有真实环境的负载测试数据，并发能力是理论估算
4. **单人瓶颈**：所有知识在一个人脑中，文档虽然全面但仍有隐性知识未外化

---

## 适用边界

这套方法论适用于：
- **1-3人团队**从零构建技术产品
- **有明确领域知识**（知道要解决什么问题）
- **AI协作开发**（人定方向+审查，AI写代码+测试）
- **时间紧迫**（周级别而非月级别）

不适用于：
- 大团队（>5人）需要更多协调机制
- 探索性项目（不知道要做什么）需要更多用户研究
- 已有大型代码库的重构（需要不同的策略）
