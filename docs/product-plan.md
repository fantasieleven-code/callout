## 一、市场洞察：一个被忽略的巨大缺口

### 当前AI编程工具生态

```
第一层：Vibe Coding（Lovable / Bolt / v0）
  → 输入描述，输出代码
  → 问题：代码质量差1.7x，45%安全漏洞，技术债默认输出

第二层：AI编码助手（Cursor / Claude Code / Copilot）
  → 人类主导，AI加速
  → 问题：没有方法论，没有架构治理，每次session从零开始

第三层：自主软件工程师（Devin）
  → AI独立完成任务
  → 问题：复杂任务只完成15%，缺乏产品判断力
```

### 被所有人忽略的缺口

```
            市面上有的              缺失的               你需要的
         ┌─────────────┐    ┌──────────────┐    ┌──────────────────┐
         │  AI写代码    │    │  AI做决策     │    │  AI造产品         │
         │  (How)      │    │  (What/Why)  │    │  (全流程)         │
         │             │ ←→ │              │ ←→ │                  │
         │ 文件级操作   │    │ 系统级思考    │    │ 产品级交付        │
         │ 任务驱动     │    │ 决策驱动     │    │ 价值驱动          │
         └─────────────┘    └──────────────┘    └──────────────────┘
         Cursor/Claude       ???                  我们的机会
```

**关键数据**：
- AI生成代码创造 **1.7x** 更多问题（CodeRabbit 2026报告）
- **45%** 的AI代码含安全漏洞（Veracode）
- 66%开发者花更多时间调试AI代码而非自己的代码
- 代码Review时间增长 **91%**（Google DORA 2025）
- **没有任何一个工具**提供方法论引导、架构治理、或多视角验证

### 一句话机会

> **市场有100个工具帮你「写代码」，0个工具帮你「做正确的决策然后写正确的代码」。**

## 二、产品定义

### 产品名：**Archon**（架构师 + 指挥官）

**一句话定位**：AI产品架构师 — 不是帮你写代码，是帮你做对决策再写对代码。

**核心价值主张**：

| 传统AI编码 | Archon |
|-----------|--------|
| 「帮我写一个登录页」 | 「你需要登录页吗？你的用户旅程是什么？JWT还是Session？先理清再动手」 |
| 从代码开始 | 从问题开始 |
| 输出：代码文件 | 输出：架构决策 + 代码文件 + 测试 + 文档 |
| 每次session孤立 | 跨session连续记忆 |
| 写完就扔给你 | 写完帮你审查（5个视角） |

### 目标用户

**主要用户**：技术创始人 / 独立开发者 / 1-3人技术团队
- 有技术背景（能看懂代码）
- 没时间做系统架构设计
- 第一次从零建产品
- 用AI写代码但「总觉得差点什么」

**次要用户**：CTO / 技术合伙人
- 带小团队（3-10人）
- 需要架构治理但没有专职架构师
- 想把自己的方法论编码化

## 三、产品架构

### 核心理念：「方法论即产品」

我们的0→1经验中，**最有价值的不是代码，是决策过程**。产品化的本质是把这个决策过程编码成可重复执行的系统。

### 3.3 产品形态选择

| 形态 | 优点 | 缺点 | 建议 |
|------|------|------|------|
| A. Claude Code插件 | 零分发成本，已有用户 | 受限于Claude Code生态 | **V1首选** |

| D. Web SaaS | 最低使用门槛 | 与本地代码脱节 | V3考虑 |

**推荐路径**：A（Claude Code插件/MCP Server）→ B（VS Code） → D（Web平台）

## 四、核心功能设计

### 4.1 方法论引擎（The Methodology Engine）

不是被动工具，是**主动引导者**。

```
用户：「我想做一个XXX产品」

Archon Step 1 自动启动：
  ├── 「你的目标用户是谁？描述3个具体使用场景」
  ├── 「用户现在怎么解决这个问题？（竞品/替代方案）」
  ├── 「你的产品和现有方案的核心差异是什么？」
  ├── 「列出你明确不做的事情（不做清单）」
  └── 输出：概念文档 + 不做清单

完成后自动进入 Step 2：
  ├── 「你有哪些现有代码/资产可以复用？」
  ├── 「技术栈偏好？（或让我根据场景推荐）」
  ├── 基于用户场景自动生成架构建议
  ├── 生成设计决策表（每个选型附理由）
  └── 输出：架构文档 + 决策表

...以此类推，7步引导到完成
```

### 4.2 多视角审查引擎（Multi-Perspective Review）

这是我们经验中最高价值的环节，也是**市面上完全没有的能力**。

## 五、商业模式

### 5.1 定价策略

| 层级 | 价格 | 包含 | 目标用户 |
|------|------|------|---------|
| **Free** | $0 | 七步方法论引导 + 3次多视角审查/月 | 试用/个人项目 |
| **Pro** | $29/月 | 无限审查 + 决策日志 + 跨session记忆 + 模板库 | 独立开发者 |
| **Team** | $99/月(5人) | Pro全部 + 团队共享决策库 + 自定义审查角色 | 小型技术团队 |

### 5.2 为什么用户会付费

```
用户心理：
  免费工具（Cursor/Claude Code）→ 写代码快了10x
  但是：
  - 「写了3天发现架构有问题，重来」 → 浪费3天
  - 「上线后发现缺了关键功能」    → 浪费1周
  - 「安全漏洞被发现」           → 声誉损失
  - 「代码越来越乱改不动」       → 技术债累积

  Archon的价值：
  $29/月 → 避免一次「方向错误导致的返工」就回本了
```

**类比**：Cursor是「加速器」，Archon是「方向盘」。加速器再快，方向错了也白搭。

### 6.2 多视角审查的Prompt工程

核心是**角色固定 + 上下文注入**：

```markdown
# CTO审查Agent Prompt（简化版）

你是一位有15年经验的CTO，审查过200+个产品的技术架构。
你的审查原则：
1. 「能不做就不做」— 每个组件都是负债
2. 「先跑起来再说」— 过度设计是创业公司第一杀手
3. 「改接口不如加字段」— 向后兼容优于完美设计

当前项目上下文：
{project_context}  ← 动态注入

当前代码结构：
{code_structure}  ← 动态注入

设计决策表：
{decision_log}  ← 动态注入

请从以下维度审查：
1. 哪些组件过度设计了？具体砍什么？省多少时间？
2. 哪些决策会在3个月后成为瓶颈？
3. 哪些「看起来简单的决策」其实有隐藏成本？
4. 如果只能再做3件事就必须上线，应该做哪3件？

输出格式：
- 🔴 必须改（阻塞上线）
- 🟡 建议改（省时间/避风险）
- 🟢 确认好（肯定做得对的决策）
```

### 6.3 核心竞争壁垒

```
可复制的：
  × 代码生成能力（所有AI工具都有）
  × 单次审查（让ChatGPT扮演CTO也行）

不可复制的（我们的壁垒）：
  ✓ 方法论引擎（7步 + 步骤间的状态流转 + 触发条件）
  ✓ 持续上下文（跨session的决策日志 + 项目记忆）
  ✓ 审查时机判断（什么时候该审查、审查什么、审查结果怎么变成TODO）
  ✓ 方法论模板库（不同产品类型的定制方法论 — 社区贡献后的网络效应）
  ✓ 实战数据（用过这套方法论构建的产品数据 — CodeLens是第一个案例）
```

## 八、风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| 「方法论」听起来太虚，用户不理解价值 | 高 | 高 | 用具体案例讲故事：「这个审查帮我砍掉了2天无用功」 |
| LLM角色扮演审查质量不稳定 | 中 | 高 | 固定Prompt模板 + few-shot案例 + 用户反馈微调 |
| Claude Code生态变化（API/政策） | 中 | 中 | 多平台适配（VS Code作为Plan B） |
| 大厂复制（Cursor/GitHub加入方法论功能） | 低 | 致命 | 速度 + 社区 + 方法论深度是壁垒 |
| 用户不愿为「不写代码的工具」付费 | 高 | 高 | 免费层做大用户基数，Pro层用「避免一次返工就回本」的ROI叙事 |

## 九、为什么是我们

1. **我们有真实案例**：不是理论方法论，是6天从零到691个测试的实战验证
2. **我们知道每一步「为什么」**：30+设计决策表不是编出来的，是一个一个踩坑踩出来的
3. **我们知道「不做什么」比「做什么」更重要**：HR SaaS专家一句话砍掉2天、CTO审查砍掉过度设计 — 这些经验编码在审查Agent里
4. **我们是自己的第一个客户**：CodeLens A+本身就是用这套方法论构建的
